# FIX-013: Implement Cloudflare Tunnel runtime management

## Summary

EDGE-001 is marked done but only config fields exist (src/hub/config.py:67-79). No runtime code starts, supervises, or manages cloudflared. docs/ops/cloudflare-tunnel.md:137 falsely claims automatic startup.

**Decision (owner-resolved):** Implement automatic tunnel management. The hub must start cloudflared as a subprocess when tunnel is enabled, supervise it, and shut it down gracefully.

**Implementation details:**
- cloudflared is installed via the official Cloudflare apt repository (signed with cloudflare-main.gpg)
- The tunnel uses a pre-provisioned token passed via environment variable `GPTTALKER_CLOUDFLARE_TUNNEL_TOKEN`
- The tunnel can be run either as a systemd service (`cloudflared service install <token>`) or as a managed subprocess (`cloudflared tunnel run --token <token>`)
- The hub should support both modes: (1) manage cloudflared subprocess directly in lifespan, or (2) defer to systemd if already installed as a service
- A separate Cloudflare API token (env var `GPTTALKER_CLOUDFLARE_API_TOKEN`) may be used for tunnel health verification via the Cloudflare API (`/accounts/{account_id}/tokens/verify`)
- **SECURITY:** Tunnel tokens and API tokens must NEVER appear in source code, tickets, logs, or documentation. Use environment variables exclusively. Redact in structured logs per AGENTS.md security rules.

## Notes

cloudflared installation is via the official Cloudflare apt repo:
- GPG key: https://pkg.cloudflare.com/cloudflare-main.gpg
- APT source: `deb [signed-by=/usr/share/keyrings/cloudflare-main.gpg] https://pkg.cloudflare.com/cloudflared any main`
- Package: `cloudflared`

Two operational modes exist:
1. **Service mode:** `cloudflared service install <token>` — systemd manages the process
2. **Direct mode:** `cloudflared tunnel run --token <token>` — hub manages the subprocess

The hub should prefer detecting an existing systemd service before falling back to direct subprocess management.
