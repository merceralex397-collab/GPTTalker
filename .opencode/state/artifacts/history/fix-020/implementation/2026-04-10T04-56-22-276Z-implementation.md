# Implementation of FIX-020: Defect 5 — MCP error double-wrapping

## File Changed

- `src/hub/transport/mcp.py`

## The Bug

The `format_tool_response` function had a success-path that consumed handler-returned error dicts (`{"success": False, "error": "..."}`) without promoting them to proper JSON-RPC 2.0 errors. This caused malformed JSON-RPC responses:

```
// Before fix - error dict gets wrapped inside result.data
{"jsonrpc": "2.0", "id": 1, "result": {"success": false, "data": {"success": false, "error": "..."}}}
```

This violates JSON-RPC 2.0 which requires errors to be in a top-level `error` key, not nested inside `result`.

## What Was Changed

Added an error-promotion check **before** the success-path handling in `format_tool_response`:

```python
# If handler returned an error dict, promote it to proper JSON-RPC error
if isinstance(result, dict) and result.get("success") is False and "error" in result:
    response["error"] = {
        "code": -32603,
        "message": result.get("error", "Unknown error"),
    }
    return response
```

This check is placed immediately inside the `else` block (where `error` param is None), before any result-wrapping logic.

## How It Produces Correct JSON-RPC 2.0 Error Shape

| Scenario | Response Shape |
|---|---|
| Handler returns `{"success": False, "error": "Not found"}` | `{"jsonrpc": "2.0", "id": ..., "error": {"code": -32603, "message": "Not found"}}` |
| Handler returns `{"success": True, "data": {...}}` | `{"jsonrpc": "2.0", "id": ..., "result": {"success": True, "data": {...}, "duration_ms": ...}}` |
| Handler returns raw value (e.g., string) | `{"jsonrpc": "2.0", "id": ..., "result": {"success": True, "data": raw_value, "duration_ms": ...}}` |

The error promotion short-circuits with `return response`, so the result is never wrapped in `result.data`. The `code: -32603` is the standard JSON-RPC 2.0 "Internal error" code.

## Verification

- Syntax check: `python3 -m py_compile src/hub/transport/mcp.py` passes
- Error dicts with `success is False` now produce `response["error"]` instead of `response["result"].data`
- Normal success cases are unaffected

## Acceptance Criteria Met

- ✅ Error dicts produce `response["error"]` as proper JSON-RPC 2.0 error
- ✅ `code: -32603` used (Internal error — appropriate for tool execution errors)
- ✅ Normal success cases still produce `{"result": {"success": True, "data": ...}}`
